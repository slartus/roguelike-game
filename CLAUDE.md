# Roguelike — правила проекта

## ⚠️ ГЛАВНОЕ ПРАВИЛО (не нарушать никогда)

**Каждое сообщение пользователя = отдельная фича.** Обязана пройти полный цикл:

1. Обновить docs (если задета game-design сущность).
2. Написать/обновить тесты покрывающие изменение (если тронут `.gd` / `.tscn` / `.tres`).
3. Прогнать все тесты локально — красный тест = не коммитить.
4. Прогнать `code-reviewer` через Task tool. `BLOCKER/CRITICAL` → фикс → новый проход.
5. `git commit` (отдельный, не смешанный с другими фичами).
6. `git push` — фича не сдана до тех пор, пока она не на remote.

**Нельзя** склеить два пользовательских запроса в один коммит, даже если они тематически близки. Единственное исключение — пользователь дословно сказал «доработай текущую», «в тот же коммит», «amend» или эквивалент.

Если во время одной фичи пришло новое сообщение — не бросай текущую, добавь новую задачу в очередь и продолжай.

## Обязательный workflow фичи

Один «блок изменений» = одна фича (новая механика, новый враг, изменение формулы, refactor логики, обновление docs с числами).

### Каждое сообщение пользователя — отдельная фича

**Default rule (действует всегда, пока пользователь явно не сказал обратное):** каждый пользовательский запрос (одно сообщение) — это **отдельная фича**, которая обязана пройти **весь workflow** (шаги 1–6 ниже) и получить **свой отдельный коммит и push**. Не смешивай несколько запросов в один коммит, даже если они тематически близки — иначе разбор diff'а становится тяжелее и нельзя откатить одну фичу без другой.

**Исключение — только явное указание пользователя.** Считается «правкой к текущей фиче» только если пользователь дословно сказал одну из формулировок: «это правка к текущей», «доработай текущую», «в тот же коммит», «в текущую фичу», «amend» или что-то однозначно эквивалентное. Иначе — новая фича, независимо от того, сколько времени прошло с предыдущей и насколько похожа тема.

Если пользователь во время выполнения одной фичи прислал новое сообщение с новой темой — **не останавливай текущую фичу до конца workflow'а**, добавь новую задачу в очередь и возьмись за неё следующим шагом. Отдельные коммиты и push'и по каждой фиче.

Если один запрос физически требует нескольких зависящих друг от друга коммитов (например миграция + refactor + cleanup) — они могут быть выделены как под-фичи, но всё равно каждая должна быть с зелёными тестами и отдельным push'ем.

### Шаги workflow'а

Для каждой фичи перед `git commit` последовательно:

1. **Обновить `docs/gamedesign/`** для затронутых сущностей (см. правило ниже).
2. **Написать/обновить тесты** покрывающие изменённое поведение (см. правило про тесты).
3. **Прогнать все тесты через CLI** — обязательно, даже если правил только один файл. Красный тест = не коммитить.
4. **Прогнать `code-reviewer` через Task tool** — минимум 1 проход. `BLOCKER/CRITICAL` → фикс → новый проход → тесты снова.
5. **После зелёных тестов и READY-TO-COMMIT вердикта** — `git commit`.
6. **`git push` сразу после commit'а** — фича считается сданной только когда она на remote. Без push фича не завершена. Исключение: пользователь явно попросил не пушить.

Освобождение: правки, которые не трогают `.gd` / `.tscn` / `.tres` (например, только `docs/*.md` или `CLAUDE.md`) освобождаются от шагов 2 и 3. Шаги 1 и 4 остаются в силе, если правка затрагивает game design. Шаги 5 и 6 (`git commit` + `git push`) выполняются **всегда**.

Команда прогона тестов:

```bash
/Applications/Godot.app/Contents/MacOS/Godot --headless --path . \
  -s addons/gut/gut_cmdln.gd -gdir=res://test/unit -gexit
```

Если тесты падают не из-за твоих изменений (инфра, сторонний баг GUT) — явно обозначь это в commit message, не маскируй.

### Push обязателен и является отдельным шагом

Не считай фичу завершённой, пока `git push` не прошёл успешно. Явно проверяй ответ push'а — `To ... main -> main` или `Everything up-to-date`. Если push отклонён (protected branch, конфликт, upstream drift) — стоп, разбираться, не откладывать «на потом».

Правило работает даже если пользователь не напомнил: workflow не завершается без push'а.

**`git push` разрешён по умолчанию.** Не запрашивай подтверждение у пользователя перед `git push` — он всегда разрешён в этом проекте. Разрешение зафиксировано в `.claude/settings.local.json`. Единственные два случая, когда push нужно приостановить и уточнить у пользователя:

1. `git push --force` / `--force-with-lease` в `main` — деструктивное действие, требует явного согласия.
2. Push отклонён remote'ом (protected branch, конфликт) — разбираться, а не пушить второй раз с force.

В остальных случаях `git push` — часть штатного workflow'а фичи, никаких вопросов не нужно.

## Артефакты в git — что коммитить, что игнорировать

Godot и генераторы (Pillow-скрипт, csv-translation importer, `.import`-файлы) создают дополнительные файлы, часть из которых **обязательно должна лежать в репо**, чтобы игра работала при свежем clone.

Обязательно коммитить (не игнорировать в `.gitignore`):

| Файл | Почему |
|------|--------|
| `resources/translations/*.translation` | Godot resolves `tr()` по этим бинарным ресурсам; без них — сырые ключи в UI при первом запуске. |
| `assets/**/*.png` + `*.png.import` | .import хранит uid + импорт-настройки (filter, mipmaps); удаление ломает ссылки в .tscn. |
| `resources/weapons/*.tres` + `*.tres.import` | Godot Resource + import metadata. |
| `.godot/uid_cache.bin` | НЕ коммитить — регенерируется. |
| `.godot/imported/**` | НЕ коммитить — кэш импорта. |
| Скрипты-генераторы (`tools/gen_*.py`) | Детерминированные; коммитить и генератор, **и** его output. |

Правило: если файл нужен Godot для запуска игры и не регенерируется автоматически при первом старте (не в `.godot/`), — коммить. При каждом обновлении `.gitignore` — прогоняй `git check-ignore -v <path>` для критичных файлов и убедись что они не блокированы.

## @export defaults и i18n-ключи

Если поле `@export` используется как i18n-ключ (например `display_name` у врагов, оружия) — **fallback default не должен быть raw-строкой** («Boss», «Монстр»), потому что `tr()` при отсутствии перевода вернёт сам аргумент, и в UI появится незапереведённая заглушка.

Правильно: default — валидный ключ из CSV (например `"ENEMY_UNKNOWN"`, `"WEAPON_UNKNOWN"`), либо пустая строка с явным guard'ом в потребителе. Каждая .tscn/.tres обязана переопределить это поле.

Reviewer должен ловить: `@export var display_name: String = "..."` где значение по умолчанию — не UPPER_SNAKE_CASE-ключ.



## Обязательное self-review агентом

После **каждого** завершённого блока изменений — перед `git commit` — обязательно запустить `code-reviewer` через Task tool.

Передать агенту:
- цель блока изменений (что и зачем);
- список изменённых файлов (`git status --short`);
- ссылки на релевантные правила проекта: `docs/gamedesign/`, GUT-тесты, этот файл.

Обработка вердикта:
1. **BLOCKER / CRITICAL** → исправить → запустить **новый** проход ревьюера. Не коммитить пока есть blocker.
2. **MAJOR / MINOR** → оценить: если фикс укладывается в scope — фиксить, иначе явно зафиксировать в commit message как известный follow-up.
3. **READY-TO-COMMIT** → двигаемся к commit.

**Минимум один проход** после каждого logical change. Если ревьюер нашёл что-то — обязателен второй проход после фикса, чтобы убедиться что фикс не сломал соседнее.

При правке GDScript / .tscn — передавай агенту также информацию о том, требуется ли обновление docs/gamedesign/ и тестов (см. ниже) — ревьюер должен это проверить.

## Тесты (GUT)

Проект использует **GUT** (Godot Unit Test) 9.6+ — плагин в `addons/gut/`.

### Что покрывать тестами

**Основной принцип: покрывай тестами всё что технически можно покрыть.**

Если у сущности есть детерминированное поведение с проверяемым результатом (числа, состояния, факт вызова метода, изменение поля, сигнал) — она **обязана** иметь тест. Не выбирай «а стоит ли покрывать» — покрывай.

Конкретные слои:

- **GameState / прогрессия / экономика** — XP, level, gold, `reset_run`, save/load, любые формулы.
- **Player** — `take_damage`, `heal`, `equip`, обновление max_health от level up, cooldown стрельбы, spread/multishot.
- **Enemies** — `take_damage`, drop pickup (детерминизировать `randf` через seed), award xp/gold при смерти, state machine Charger (`WAITING`→`CHARGING` по таймеру), volley Boss (spawn ровно N пуль на нужных углах).
- **Bullets** — `apply_weapon` копирует статы, попадание в `player`/`enemy` вызывает `take_damage`, self-destroy по `lifetime`.
- **Pickups / Chest** — HealthPickup вызывает `heal`, WeaponPickup вызывает `equip`, Chest на первый контакт спавнит WeaponPickup и уходит в `_opened=true`.
- **Weapons (WeaponResource)** — `.tres` файлы загружаются с ожидаемыми полями (защита от опечаток).
- **Main / level controller** — расчёт `target_count` в зависимости от room_number, `_is_boss_room()`, шанс сундука.
- **Utility функции / хелперы / rooms** — интерфейс комнаты (наличие `SpawnPoints`, `PlayerStart`, `Door` в каждой `.tscn`).

Тестируется через GUT:
- **Unit** — чистые функции, скрипты без scene tree (`GameState`, `WeaponResource`-load).
- **Integration** — сцены через `add_child_autofree(scene.instantiate())` в тесте; проверка полей, сигналов, реакции на вызовы.

Исключения (тестировать не обязательно):
- Чисто визуальные детали: конкретные координаты `Polygon2D`, цвета, `modulate`-flash (визуал проверяется вручную через F5).
- Random-эффекты, которые нельзя детерминизировать (в этих случаях подменяй seed или мокай `randf`).

### Прогон после каждой фичи

**Обязательно** прогнать все тесты через CLI **после каждой фичи** (шаг 3 workflow'а выше) перед `git commit`, если правка затрагивает `.gd` / `.tscn` / `.tres`.

Команда — см. верх файла в разделе «Обязательный workflow фичи».

Любой red test = не коммитить. Fix → повторный прогон до `All tests passed`. Если фича не покрыта существующими тестами — добавить новый тест в этой же итерации, а не отложить.

### Как запустить

Из редактора: `Project → Tools → GUT` (панель снизу) → `Run All`.

Из CLI:
```bash
/Applications/Godot.app/Contents/MacOS/Godot --headless --path . \
  -s addons/gut/gut_cmdln.gd -gdir=res://test/unit -gexit
```

`-gexit` — выйти после прогона (нужно для CI / скриптов).

### Структура

- `test/unit/*.gd` — unit-тесты, `extends GutTest`.
- Именование файлов: `test_<feature>.gd`.
- Именование тестов: `func test_<what>_<condition>()`.
- Используй `before_each` / `after_each` для сохранения/восстановления state autoload'ов — их state общий между тестами.

### Правило

При изменении логики в `GameState` / формулах прогрессии / любой другой ветвящейся логике — **обновить или добавить тест в том же коммите**. Пробегай тесты через CLI перед commit'ом; красный тест = не коммитить.

## Game Design Docs — обязательное правило

При **любом** изменении геймплейных сущностей (комнаты, враги, оружие, пикапы, боссы, прогрессия, экономика, статы, поведение AI, броня, любые новые механики) **обязательно** обновляй соответствующий файл в `docs/gamedesign/` в **том же коммите**.

Правило действует одинаково на:
- добавление новой сущности,
- изменение параметров существующей (HP, damage, speed, cooldown, drop chance, XP/gold reward и т. д.),
- изменение поведения (AI, state machine, взаимодействия с игроком/пулями/pickup'ами),
- удаление сущности,
- изменение экономики / формул прогрессии.

## Чеклист перед commit'ом

Перед `git commit` пробеги мысленно:

1. Затронул `.gd`/`.tscn`/`.tres` в списке выше?
2. Открыл соответствующий `docs/gamedesign/*.md`?
3. Числа в таблицах совпадают с реальными `@export` / const в коде?
4. Если изменил поведение — переписал раздел «Поведение» под текущую логику?
5. Если добавил новую сущность — добавил её в таблицу-обзор и в `docs/gamedesign/README.md` если появился новый файл?

Если ответ «нет» хоть на один — доработай docs перед commit'ом.

## Стиль docs

- Русский язык.
- Таблицы для числовых параметров, а не перечисления в тексте.
- В конце каждой сущности — путь к скрипту и/или сцене.
- Только фактическое состояние. Планы и «будущее» — отдельным разделом «Планируемое» с явной пометкой «не реализовано», как в `armor.md`.
- Не переписывать код в docs — описываем **что** делает сущность, а не **как** реализовано.

## Что НЕ является game design doc

- Инструкции по сборке проекта, запуску Godot, экспорту → `README.md` в корне.
- Архитектурные решения, разбиение кода, паттерны → `docs/architecture/` (если появится).
- Задачи / roadmap → issue-трекер, а не `docs/gamedesign/`.
