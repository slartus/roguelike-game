# Workflow фичи

Одна фича = отдельная ветка + отдельная worktree-папка + отдельный PR в `main`. В `main` напрямую не коммитим, в основном клоне репозитория не работаем.

Что считать одной фичей — определяется по смыслу задачи, а не по количеству сообщений. Несколько сообщений подряд могут доводить одну фичу до готовности — работаем в уже созданной ветке/папке. Новая ветка/папка нужна, когда пользователь явно переключает контекст («новая задача», «отдельно сделай X», «начнём другую фичу») либо когда я вижу, что задача не связана с текущей — тогда уточняю у пользователя.

## Изоляция фичи: worktree под каждую задачу

Одна папка репозитория = один `HEAD`. Если 2+ Claude-сессии работают в одном каталоге, каждая делает `git checkout <своя_ветка>` и `HEAD` прыгает под другими сессиями. Файлы других сессий восстанавливаются из ветки, работающие Edit'ы теряются, staging скидывается, коммиты уходят в reflog.

Решение — под **каждую** фичу отдельный git worktree в отдельной папке-соседке:

```bash
MAIN_REPO=/Users/artemslinkin/projects/roguelike-game
git -C $MAIN_REPO fetch origin
git -C $MAIN_REPO worktree add ../roguelike-<slug> -b <type>/<slug> origin/main --no-track
cd ../roguelike-<slug>
git branch --unset-upstream  # safety, --no-track уже отвязал tracking
```

- `<slug>` — kebab-case, тот же, что и в имени ветки.
- Дальнейший workflow (docs → tests → run tests → self-review → commit → push → PR) выполняется целиком внутри `../roguelike-<slug>`.
- Основной каталог `roguelike-game` не трогаем — там может параллельно работать другая Claude-сессия либо сам пользователь в IDE.
- После merge PR — `cd $MAIN_REPO && git worktree remove ../roguelike-<slug>` и `git branch -d <type>/<slug>`.

### Никогда не делать `git checkout <существующая_ветка>` в общем каталоге

В общем каталоге `git checkout <branch>` меняет `HEAD` и working tree для **всех** сессий, работающих через этот каталог. Параллельная сессия увидит, что её файлы «перешли» на чужую ветку, а её незакоммиченные правки могут пропасть.

Разрешено:
- Создать НОВУЮ ветку и НОВУЮ папку через `git worktree add ... -b <type>/<slug> origin/main`.
- Работать внутри своей worktree-папки без переключений.

Запрещено:
- `git checkout main`, `git checkout <foreign_branch>`, `git switch <branch>` в основном каталоге.
- Даже `git checkout <моя_ветка>` не нужен — я и так на ней с момента `git worktree add -b`.

### Если `HEAD` внезапно оказалась не на моей ветке

Признак того, что в этой же папке параллельно работает другая Claude-сессия либо пользователь в IDE. `git checkout` обратно на свою ветку не решает — только раззадоривает race. Правильные действия:
1. Остановиться, не делать новых Edit'ов.
2. Явно сказать пользователю: HEAD ушла на `<other_branch>`, кто-то ещё работает в этом каталоге.
3. Спросить, продолжать ли в другой worktree-папке или дождаться.

## 8 обязательных шагов перед считать фичу сданной

1. **Branch + worktree** — новая ветка `<type>/<slug>` от актуального `origin/main` через `git worktree add ../roguelike-<slug> -b <type>/<slug> origin/main --no-track` (см. «Изоляция фичи» выше). Все дальнейшие шаги — внутри `../roguelike-<slug>`.
2. **Docs** — обновить `docs/gamedesign/*.md` если задета game-design сущность (см. `20-gamedesign-docs.md`).
3. **Tests** — написать/обновить тесты покрывающие изменённое поведение (см. `10-tests.md`).
4. **Run tests** — прогнать всю GUT-сюиту через CLI. Красный тест = не коммитить.
5. **Self-review** — прогнать `godot-code-reviewer` через Task tool (см. `.claude/agents/godot-code-reviewer.md`). `BLOCKER/CRITICAL` → фикс → новый проход → тесты снова.
6. **Commit + push ветки** — отдельный коммит на фичу, не смешанный с другими. Push ветку в `origin` (`git push -u origin <branch>` первый раз).
7. **PR в main** — открыть PR через `gh pr create --base main`. Без PR фича не сдана.
8. **Cleanup после merge** — как только PR смержен (через GitHub UI пользователем либо `gh pr merge` по явной просьбе), **обязательно** удалить worktree-папку и локальную ветку (см. раздел «Cleanup после merge» ниже). Оставленные worktree'ы копятся в `../roguelike-*` и в `git worktree list`, работают в устаревшем состоянии и путают следующие сессии.

Быстрый запуск всего цикла: skill `/feature-close`.

## Правило «что считать одной фичей»

Одну и ту же фичу можно доводить несколькими сообщениями подряд — работаем в уже созданной ветке/папке, не заводим новую при каждом сообщении.

Новая ветка/папка нужна когда:
- пользователь явно сказал «новая задача» / «отдельно сделай X» / «начнём другую фичу»;
- задача явно не связана с текущей (другая подсистема, другая цель) — тогда спрашиваю пользователя, продолжать в текущей ветке или заводить новую;
- пользователь дал новую задачу после того, как я уже закрыл PR текущей фичи.

Если во время работы над фичей приходит новое сообщение по другой теме — не бросай текущую, добавь новую в очередь через `TaskCreate`, доведи текущую до PR, потом заводи новый worktree под следующую (стартуя от свежего `origin/main`).

Если один запрос физически требует несколько зависящих коммитов (миграция → refactor → cleanup) — каждый под-коммит проходит все 8 шагов и уезжает своим PR (stacked PR: base предыдущего PR, а не `main`, если они действительно цепляются) в своей worktree-папке.

## Ветка и PR

### Имя ветки

Формат: `<type>/<slug>`, где `type ∈ {feat, fix, refactor, docs, chore, test}`, `slug` — kebab-case, короткое английское описание фичи.

Примеры: `feat/melee-arc-swing`, `fix/hud-gold-key`, `refactor/weapon-controller-cooldown`, `docs/upgrades-overview`.

Ветку и worktree-папку создаём одной командой (см. «Изоляция фичи»):

```bash
MAIN_REPO=/Users/artemslinkin/projects/roguelike-game
git -C $MAIN_REPO fetch origin
git -C $MAIN_REPO worktree add ../roguelike-<slug> -b <type>/<slug> origin/main --no-track
cd ../roguelike-<slug>
```

**Не** делаем `git checkout main; git pull; git checkout -b <branch>` в основном каталоге — это меняет `HEAD` для всех, кто работает там. `git worktree add -b ... origin/main` создаёт ветку от свежего remote без переключения `HEAD` основного каталога.

### Push ветки

Первый push — с `-u`, чтобы upstream был на одноимённую remote-ветку, а не на `main`:

```bash
git push -u origin <type>/<slug>
```

Последующие пушы в эту же ветку — просто `git push` (из worktree-папки).

**Никогда не пушить в `main` напрямую.** Если случайно оказался на `main` с коммитом (не должно случаться, если следовать worktree-flow) — переносим коммит на новую ветку через worktree и делаем стандартный push веткой.

### PR через gh

```bash
gh pr create --base main --head <type>/<slug> \
  --title "<type>(<scope>): <краткое описание>" \
  --body "$(cat <<'EOF'
## Что

<1–3 предложения по существу>

## Зачем

<опционально: мотивация / контекст>

## Как проверять

- [ ] `/run-tests` — GUT-сюита зелёная
- [ ] визуальный smoke-test если задета механика игрока
EOF
)"
```

После `gh pr create` — вернуть URL PR'а пользователю.

### Что делать если PR не проходит CI / ревью

- CI красный — фиксить в той же ветке (в той же worktree-папке), коммитить, `git push`. PR обновится автоматически.
- Reviewer запросил правки — те же правки в ту же ветку. Не открывай новый PR.
- Ветка разошлась с `main` (конфликт merge) — `git fetch origin && git rebase origin/main` (из worktree-папки), решить конфликты, `git push --force-with-lease` (только на свою фиче-ветку, никогда на `main`).

### Merge

По умолчанию **не мержим PR из-под агента** — это делает пользователь через GitHub UI после ревью. Исключение: пользователь явно попросил «смерджи» / «замерджи PR». Тогда — `gh pr merge <number> --squash --delete-branch` (или `--merge` если пользователь предпочитает).

### Cleanup после merge (шаг 8)

Как только PR смержен, **обязательно** прибраться. Оставленные worktree'ы:
- копятся в `../roguelike-*` и в `git worktree list`;
- застревают на устаревших ветках, которых уже нет в `origin`;
- путают следующие Claude-сессии («что за папка `../roguelike-hud-icons`? актуальна ли?»);
- забивают diff в IDE (файл-индексаторы обходят все worktree'ы).

Порядок:

```bash
cd $MAIN_REPO
git fetch --prune origin                        # подхватить удаление ветки в origin
git worktree remove ../roguelike-<slug>         # удалить папку
git branch -d <type>/<slug>                     # удалить локальную ветку
```

Если worktree содержит незакоммиченные правки (не должно случаться после merge, но проверим) — `git worktree remove` выдаст ошибку. Тогда сначала выяснить, что за правки, и либо сохранить в новом worktree, либо использовать `--force`.

Проверить что всё чисто: `git worktree list` показывает только `$MAIN_REPO`, `git branch --list '<type>/<slug>'` пустой.

Если пользователь мержит PR молча (сообщил уже после факта или сам увидел в UI) — при следующей активности проверить статус смерженных PR через `gh pr list --state merged --limit 5` и почистить worktree'ы для смерженных фич.

## Освобождения

Только `docs/*.md` / `CLAUDE.md` / `.claude/**` изменены и `.gd`/`.tscn`/`.tres` не тронуты → шаги 3 и 4 (tests, run tests) можно пропустить. Шаги 1, 2, 5, 6, 7, 8 остаются — даже docs-only фичи едут через свою ветку, свою worktree-папку и свой PR, и убираются после merge.

## Push и safety

`git push` на фиче-ветку — часть штатного workflow'а, разрешён по умолчанию (`.claude/settings.local.json`).

Останавливаюсь и уточняю только:
1. `git push --force` / `--force-with-lease` в `main` — деструктивно, нельзя.
2. `git push` в `main` напрямую (минуя PR) — нарушает главное правило, нельзя.
3. Push отклонён remote'ом (protected / конфликт на фиче-ветке) — разбираться, не делать force «на автомате».
4. `HEAD` внезапно на чужой ветке — стоп, признак параллельной сессии в общем каталоге (см. «Изоляция фичи»).

Проверяю ответ push'а — `To ... <branch> -> <branch>` или `Everything up-to-date`. Пока не увидел успех — PR не открывается.

Проверяю ответ `gh pr create` — должна быть URL вида `https://github.com/<owner>/<repo>/pull/<n>`. Пока не увидел — фича не сдана.
